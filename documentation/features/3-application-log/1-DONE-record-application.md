# 1. DONE - Record Application

## What it does

Provides the append-only write path for the application log — an MCP tool that records one entry per `/apply`
attempt (URL, a server-generated timestamp, and the outcome) to a local, plain-text log file. This is the only
way an attempt's outcome gets persisted; nothing else in the system writes to this file.

## Inputs / outputs

- Input: `record_application(url: str, outcome: Literal["success", "aborted", "stuck-unresolved"])`.
- Output: on success, no return value needed beyond acknowledgement (e.g. `{"ok": true}`); on failure to write
  (e.g. disk full, permissions), returns a structured error the LLM can see and report to the user.
- Storage format: one JSON object per line (JSON Lines) appended to a fixed local file (e.g.
  `~/.peddler/applications.log`), each line `{"url": ..., "timestamp": "<ISO 8601 UTC>", "outcome": ...}`.
  Timestamp is generated by the tool itself at call time — never passed in by the caller — so the log can't be
  backdated by a mistaken LLM-supplied value.

## Edge cases and failure modes

- Log file/parent directory doesn't exist yet on first run → created automatically.
- Concurrent writes (should not happen given the "one active session at a time" constraint from
  `Requirements.md → Non-Functional → Concurrency`, but the append must still be a single atomic write — e.g. one
  `write()` call of a complete, newline-terminated line — so a crash mid-write can't corrupt a previous entry).
- Disk full / permission denied on write → tool returns a structured error rather than raising an unhandled
  exception that kills the MCP server process.
- `outcome` value outside the three allowed literals → tool rejects with a validation error rather than writing
  malformed data.

## Acceptance criteria

- Calling `record_application` with a valid URL and one of the three allowed outcomes appends exactly one
  well-formed JSON line to the log file, including a timestamp the tool generated itself (not caller-supplied).
- The log file is created automatically (including parent directories) if it doesn't already exist.
- Calling it with an invalid `outcome` value returns an error and does not write anything to the file.
- Simulating a write failure (e.g. read-only file) returns a structured error result rather than crashing the
  process.
- Two sequential calls result in two lines, each independently valid JSON — no line is ever partially written or
  corrupted by a subsequent call.

## Tasks

1. Create `src/peddler/applog/store.py` with log-file path resolution and an `append(url, timestamp, outcome)`
   JSON Lines writer (single atomic `write()` per call, auto-create parent directory).
2. Create `src/peddler/applog/tools.py` with the `record_application` MCP tool wrapper: validates `outcome`
   against the three allowed literals, generates the timestamp, calls the store, wires into the
   `1-mcp-server-framework` registry.
3. Write `tests/applog/test_store.py` (append/format/atomicity/error-handling) and `tests/applog/test_tools.py`
   (valid call, invalid outcome, simulated write failure) covering all five acceptance criteria.

## Deliverables

- `src/peddler/applog/store.py` — log file path resolution, JSON Lines append logic.
- `src/peddler/applog/tools.py` — the `record_application` MCP tool wrapper, registered against the framework
  from `1-mcp-server-framework`.
- `tests/applog/test_store.py` — append/format/atomicity/error-handling tests for the storage layer.
- `tests/applog/test_tools.py` — tool-level tests (valid call, invalid outcome, simulated write failure).

## Dependencies

`1-mcp-server-framework` (tool registration/dispatch).
